# LR13 — Engineering Checklist

Короткий чек-лист после лекции.
Его можно использовать перед сдачей лабораторной, перед открытием PR и перед самостоятельной проверкой проекта.

## 1. Границы задачи

Проверьте:

- целевая работа лежит в ожидаемой папке курса;
- в проекте понятно, где основная реализация;
- в PR нет случайных файлов из других лабораторных;
- структура каталогов помогает проверке, а не мешает ей.

Ответьте на вопросы:

- если другой человек откроет проект, он быстро поймёт, где находится нужная работа?

## 2. Контракт между слоями

Проверьте:

- frontend, route, schema и service используют одни и те же имена полей;
- endpoint отражает реальную операцию;
- один и тот же сценарий не называется по-разному на разных слоях;
- route не валидирует одно, а сервис не использует другое.

Ответьте на вопросы:

- одинаково ли понимают данные все части системы?
- не расходятся ли имена, формат и смысл payload?

## 3. Авторизация и ownership

Проверьте:

- наличие токена не подменяет проверку прав;
- для операций с ресурсом проверяется владелец;
- по одному только `id` чужой ресурс нельзя изменить или завершить;
- mock-режим не выдается за готовую боевую реализацию.

Ответьте на вопросы:

- кто может работать с этим ресурсом?
- где именно это проверяется в коде?

## 4. Тесты

Проверьте:

- тесты страхуют реальный риск, а не только happy path;
- есть тесты на основной сценарий, а не только на `401` и `health`;
- тесты покрывают важные стыки: контракт, ownership, валидацию, submit flow;
- test scripts действительно запускаются на чистом окружении.

Ответьте на вопросы:

- какой самый неприятный сбой может произойти?
- есть ли тест, который это поймает?

## 5. Воспроизводимость

Проверьте:

- проект запускается по понятному сценарию;
- зависимости и команды явно описаны;
- локальная среда не является скрытым обязательным условием;
- нет неявной зависимости на вашу локальную БД, кэш или временные файлы.

Ответьте на вопросы:

- можно ли воспроизвести запуск на другой машине без ваших личных обходных путей?

## 6. Чистота PR

Уберите из PR:

- `dev.db` и другие локальные БД;
- `coverage`;
- `.vite`;
- `trace.zip`;
- backup-каталоги;
- временные и служебные файлы, не относящиеся к задаче.

Ответьте на вопросы:

- если смотреть только на diff, понятно ли, что именно вы сделали по задаче?

## 7. Признаки зрелого проекта

Можно считать, что проект стал заметно зрелее, если:

- у него ясная структура;
- у него единый контракт между слоями;
- у него есть ownership-проверка ресурсов;
- тесты ловят реальные риски;
- запуск воспроизводим;
- PR чистый и ограничен по цели.

## Мини-чек перед сдачей

Перед отправкой PR быстро пройдитесь по этим вопросам:

1. Работа лежит в правильной папке?
2. Слои используют один и тот же контракт?
3. Проверяется ли доступ к конкретному ресурсу?
4. Тесты ловят реальный риск?
5. Проект воспроизводим на чистом окружении?
6. В PR нет локального мусора?

Если на один из этих вопросов ответ "нет" или "не уверен", значит перед сдачей ещё есть что улучшить.
